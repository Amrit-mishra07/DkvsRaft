DISTRIBUTED KEY-VALUE STORE WITH RAFT CONSENSUS
Architecture, Design, & Implementation Guide
================================================================================

1. THE PROBLEM DOMAIN & CAP THEOREM
--------------------------------------------------------------------------------
A standard single-node key-value store represents a Single Point of Failure (SPOF)
and a bottleneck for scalability. In the context of the CAP theorem (Consistency, 
Availability, Partition Tolerance), we aim to build a CP system: a strongly consistent 
data store that remains available as long as a majority of nodes are reachable, 
fully tolerating network partitions.

To achieve this, data must be replicated across multiple machines. However, replication
introduces the complex problem of Distributed Consensus: how do multiple independent, 
potentially failing machines agree on the exact sequence of state changes?

We solve this using State Machine Replication (SMR) powered by the Raft Consensus 
Algorithm. Raft is the industry standard for consensus, underlying major infrastructure
like etcd (Kubernetes), Consul, and CockroachDB. This project demonstrates practical 
knowledge of cluster orchestration, fault tolerance, and gRPC communication.


2. SYSTEM ARCHITECTURE
--------------------------------------------------------------------------------
The system is decoupled into two primary layers, enforcing a clean separation of 
concerns:

  LAYER 1: The Consensus Engine (Raft)
  This layer handles the replication log, peer-to-peer gRPC communication, and 
  maintaining cluster authority. It treats all client requests as opaque byte 
  commands. It provides three absolute guarantees:
    - Election Safety: At most one leader can be elected in a given term.
    - Log Matching: If two logs contain an entry with the same index and term, 
      the logs are identical in all entries up through that index.
    - State Machine Safety: If a node has applied a log entry to its state 
      machine, no other node will ever apply a different command for that index.

  LAYER 2: The Application State Machine (KV Store)
  A deterministic state machine (a hash map) residing on each node. It blindly 
  executes commands (PUT, GET, DELETE) only when the Consensus Engine notifies it 
  that a command has been formally "committed" across the cluster quorum.


3. CORE MECHANISMS
--------------------------------------------------------------------------------
A. Leader Election & Split-Brain Mitigation
   The cluster operates using a strong leader model. Only the leader handles client 
   writes. If followers stop receiving heartbeats, they transition to Candidates, 
   increment the logical clock (Term), and request votes. 
   Crucially, Raft prevents "Split-Brain" scenarios (two leaders active at once) 
   by requiring a strict mathematical majority (Quorum) to win an election. Since 
   any two majorities must intersect by at least one node, conflicting leaders 
   cannot emerge.

B. Log Replication & Durability
   Writes are not instantly applied. The leader appends the command to its log and 
   broadcasts `AppendEntries` RPCs. A command is only marked as "Committed" once 
   the leader receives successful acknowledgments from a quorum. Once committed, 
   the system guarantees the data will survive even if the leader crashes.


4. LIFECYCLE OF A WRITE OPERATION
--------------------------------------------------------------------------------
  1. Routing: A client sends `PUT x=1` to the cluster. If it hits a follower, 
     the follower redirects the client to the current leader.
  2. Proposal: The leader appends the command to its local log as uncommitted.
  3. Replication: The leader fires concurrent `AppendEntries` gRPC calls to followers.
  4. Quorum Acknowledgment: Followers append the entry to their logs and reply `Success`.
  5. Commit: Once a majority (e.g., 2 out of 3) acknowledges the entry, the leader 
     advances its `commitIndex`.
  6. Application: The leader applies `PUT x=1` to its KV store and replies to the client.
  7. Follower Sync: On the next heartbeat, the leader informs followers of the new 
     `commitIndex`, prompting them to apply the command to their own KV stores.


5. FAULT TOLERANCE CAPABILITIES
--------------------------------------------------------------------------------
The system uses `2f + 1` architecture. A 3-node cluster tolerates `f=1` failure. 
A 5-node cluster tolerates `f=2` failures.
- Leader Crash: A follower's election timer expires, triggering a seamless election.
- Network Partition: If a leader is isolated, it cannot achieve a quorum for new 
  writes and simply stalls (preserving consistency). The majority partition will 
  elect a new leader and continue processing.


6. ENGINEERING SCOPE LIMITATIONS (MVP vs PRODUCTION)
--------------------------------------------------------------------------------
To deliver a high-impact, focused project, certain production features are explicitly 
scoped out. Being able to explain *why* these are excluded is a strong interview signal:
- Log Compaction/Snapshotting: Excluded to focus on core replication mechanics. Logs 
  will grow indefinitely in this MVP.
- Dynamic Membership: Cluster size is static at startup. Adding/removing nodes at 
  runtime requires complex joint-consensus logic.
- Disk Persistence: The Write-Ahead Log (WAL) is kept in-memory for simplicity. A 
  crash of the *entire* cluster simultaneously will result in data loss.


7. IMPLEMENTATION ROADMAP (MILESTONES)
--------------------------------------------------------------------------------
Phase 1: Cluster Orchestration & Leader Election
  - Define Node states (Follower, Candidate, Leader) and gRPC interfaces.
  - Implement randomized election timers and heartbeats to prevent election collisions.
  - Implement `RequestVote` RPC logic, enforcing up-to-date log checks before voting.
  - *Validation: Kill the leader container and observe a new leader taking over.*

Phase 2: State Machine Replication (Log Matching)
  - Implement the `AppendEntries` RPC for log replication.
  - Implement the Log Matching Property: followers reject entries if their previous 
    log terms/indexes do not match the leader's.
  - Track `commitIndex` on the leader based on majority consensus.

Phase 3: The Application State Machine (KV Store)
  - Build the deterministic Key-Value hashmap.
  - Create a background pipeline that listens for `commitIndex` advancements and 
    applies commands to the hashmap in strict sequential order.

Phase 4: Client API & Cluster Routing
  - Expose a REST or gRPC client API for `GET`, `PUT`, `DELETE`.
  - Implement routing: Followers must identify the current leader and forward or 
    redirect client traffic.
  - Ensure Linearizable Reads by ensuring the leader verifies its status with a 
    heartbeat before serving a `GET`.

Phase 5: Resilience Testing & Chaos Engineering
  - Build automated tests simulating network drops, delayed RPCs, and process crashes.
  - Demonstrate recovery of uncommitted logs when a partitioned leader rejoins the 
    main cluster.
